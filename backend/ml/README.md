# Step-role classifier

A tiny local model that labels each step profile's **role** — `router`,
`retriever`, `extractor`, `generator`, or `creative` — from its system-prompt
embedding. The role tells the anomaly engine what *kind* of step it is, which sets
the fence width (`variance → k`), gates which checks run, and seeds cold-start
baselines. It runs once per new step profile, reusing the embedding the
fingerprinter already computes, so it adds zero per-call cost.

How it works: the system prompt is embedded with `all-MiniLM-L6-v2` (the same
model the fingerprinter uses) into a 384-dim vector, and a multinomial
`LogisticRegression` maps that vector to a role. It's trained on the labelled
prompts in `step_training_data.jsonl`.

## Files

| File | What it is |
|---|---|
| `step_training_data.jsonl` | the labelled data — one `{"system_prompt", "role"}` per line. **The source of truth for which roles exist.** |
| `train_step_classifier.py` | embeds the data, fits the model, reports accuracy, saves the artifact |
| `step_classifier.joblib` | the trained model (~16 KB) — regenerated by the train script, not edited by hand |
| `try_classifier.py` | CLI to classify your own prompts |
| `../services/step_classifier.py` | production inference (`classify_role(embedding)`) |

## Prerequisites

Run everything from the `backend/` directory. The `.venv/bin/python` prefix uses
the project virtualenv (deps: `scikit-learn`, `joblib`, `numpy`,
`sentence-transformers`, all already in `requirements.txt`). The first run
downloads the ~80 MB MiniLM model and caches it.

```bash
cd backend
```

## Commands

### Classify your own prompts
```bash
# one or more prompts (quote each):
.venv/bin/python -m ml.try_classifier "Summarize this contract" "Write a haiku about rain"

# interactive — type a prompt per line, Ctrl-D to quit:
.venv/bin/python -m ml.try_classifier
```
Each line prints the decision (`role (variance=…)`, or `DECLINED` when the top
class is under the 0.40 confidence floor) plus the full probability spread across
all roles, so you can see how close the call was.

### Retrain and measure accuracy
Run this after editing `step_training_data.jsonl`:
```bash
.venv/bin/python -m ml.train_step_classifier
```
Prints the **5-fold cross-validated accuracy** and a per-role precision/recall/f1
table, then overwrites `step_classifier.joblib`.

### See the confusion matrix
```bash
.venv/bin/python -m ml.train_step_classifier --confusion
```
Adds a matrix (rows = true role, cols = predicted) so you can see *which* role
gets mistaken for which — that's where to aim more training examples.

## The improvement loop

1. Poke the model (`try_classifier`) or read the confusion matrix to find a weak
   boundary (e.g. `generator` leaking into `router`).
2. Add sharp, unambiguous examples for that boundary to
   `step_training_data.jsonl` — prompts that clearly belong to one role.
3. Retrain (`--confusion`) and watch the numbers.
4. Repeat. Accuracy climbs fast at first, then plateaus (mid-90s is the practical
   ceiling for this embedder); once there, stop tuning and wire the role up.

## Adding or removing a role

Roles are defined implicitly by the distinct `"role"` labels in the JSONL — there
is no separate enum. To add one, add labelled examples with the new label and
retrain; the model picks it up automatically. **But:** a new role should earn its
place by unlocking a *distinct check* — if it just duplicates an existing role's
checks, it only adds confusion (we tried `evaluator` and it dragged `router` down
8 points, so we dropped it). Also update `ROLE_VARIANCE` in
`../services/step_classifier.py` so the new role maps to a variance level.

## Notes

- **Uncertainty is safe.** Below the 0.40 confidence floor `classify_role` returns
  `None` rather than guessing — the caller falls back to the default fence. A
  missing artifact does the same. Roles never silently force wrong behaviour.
- The 16 KB artifact is committed so the app runs without retraining; regenerate
  it whenever the training data changes.
